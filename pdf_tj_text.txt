Overview
WorkForce Software\222s REST APIs enable customers to send to and receive data from the 
WorkForce Integration platform. Client developers can set up a seamless integration between 
their own systems and the WorkForce Suite. Our modern, advanced APIs use common
ly 
understood technology and techniques so the WFS platform can be easily adopted into your 
API ecosystem.
Getting Started
The following sections provide the information needed to easily get started in using the 
WorkForce Software REST APIs.
Prerequisites
Your account must meet the following criteria before getting started with using WorkForce 
REST APIs: 
WorkForce Software v20.2 or later
WorkForce Suite enabled
Connection to the OCI data center \(data feed only\
This requires the Data Feeds API feature flag to be enabled, which must be 
requested through a 
Salesforce ticket

Swagger UI
web document at

. The specifications provide definitions 
of all WorkForce public APIs. If you are experienced with REST APIs, the OpenAPI spec may be 
all you need. If you need further information, e.g., how to achieve certain outcomes, then this 
document can provide more 
context surrounding API use.
Authentication and Authorization
You will authenticate all API calls made over HTTPS using OAuth 2.0 protocols. 
To call an API to retrieve \(GET\ or push \(PUT\ data, you must first call the Auth API using your 
Secret Key. This retrieves an
Access Token
which is then used in an HTTP GET or PUT request to 
the API. Data is then returned in JSON format as specified for each API. 
You must request a Secret Key from WorkForce Support via Salesforce.
Secret keys are issued 
for an associated client id. Each client id can be configured for single
role
or a 
group of roles.

Behavior of REST interfaces
WorkForce APIs follow regular HTTP verb semantics.  
PUT replaces the resource referenced in the request with the resource contained within the 
request body. PUT messages are for a single instance of an entity; the Person API takes one 
Person per request.
It is used to both create and update the state of a resource on the server. 
For example, sending details of a person who does not exist in the system to the Person API will 
result in a new employee/user. If the person does exist in the system, then the det
ails we 
currently hold are replaced by the ones you send. Where date effective records are supported, 
it is possible to send just the latest data and not be required to always send full history. 
GET is used to retrieve data and does not involve any changes to data held in the system. A call 
to a resource without any parameters will return all data \(up to a max\. The cursor parameter is 
used to indicate where in the data stream to retrieve from. The count parameter is used to

GET and PUT methods are safe to repeat with the same payload. Concurrent requests to

API Versions
The WorkForce REST APIs versions are long lived. As further features are added to the APIs, this 
is always done in such a way that is backwards compatible, unless a breaking change is 
unavoidable. Breaking changes will be released as a new version. Concurr
ent versions will be 
supported with older versions eventually being deprecated and ultimately removed as 
appropriate and in consultation with customers. 
Data Feed APIs
Data Feed API are only available to customers who have been implemented in an OCI data 
center. If you have not yet been migrated, this is a pre
requisite of using Data Feeds.
Data Feed APIs are intended to be called frequently to supply real
time data. These APIs return 
an array of JSON objects, each with an identifier, along with a cursor to identify how far along 
the data stream the API client has reached. Data Feed APIs allo
w the cursor as an optional 
parameter. However, in normal use, clients are expected to provide the cursor returned from 
the previous call, allowing the client to continuously receive a data feed as it is modified or 
created by WorkForce Software applicatio
ns.
The Data feed API feature flag must be enabled. There must be a separate request for each 
environment.
Asynchronous Processing APIs
APIs will respond immediately with any general problems with the request. There are also more 
detailed checks that occur after the API request has been received and a response sent. For 
example, a data validation 
failure, which the client needs to know about. As the request is 
completed, the API responds with the HTTP response codes shown earlier. To receive details of 
any errors generated during asynchronous processing, developers must supply an email 
address. An 
error reporting mechanism which supports other channels besides email is on our 
road map.

Create a Data Store
Overview
The WorkForce data feed APIs are designed to give near real
time access to your data. These 
data feeds utilize a cursor concept allowing clients to pull data in the order in which it was 
updated or deleted in WorkForce, keeping track of where you are in th
e stream of events using 
the cursor. A call to the API with no 
cursor
parameter being passed will result in the earliest 
record updates being returned. e.g., the earliest update to an employee. The number of records 
returned is controlled by the 
count
parameter, 2000 records by default, with a maximum of 
10,000 records. 
The APIs have been kept as simple as possible, such that in most cases the data from the API 
can be stored in at most 2 tables e.g., a main domain object table for attributes which are not 
date effective and a linked table for those that are. Where data du
plication is not a concern or a 
convenient reporting feature, data can often be stored in a single table if that is preferred but 
will require an appropriate composite primary key \(PK\ to be selected. The model described 
below tries to strike a balance, in
most cases avoiding duplication. Note that there are objects 
in the model where no PK is marked. 
This happens where it is assumed data will always be 
queried as part of a join with a table which does have a PK. The model is only a recommended 
starting point and is open to extension and modification to suit different customer use cases. 
For example, th
ere are many fields in Employee that will not be used by many customers, and 
these can obviously be omitted. 
Before You Get Started
